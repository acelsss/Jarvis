## Update 1：REPL 调试输出增强

### 目的与要点

- 在 REPL（`scripts/chat.py`）中增加调试可视化输出，方便在终端查看：
  - 每一轮命中的记忆内容和分数
  - 发给 DeepSeek 的最终 messages（prompt）内容
- 不改动现有业务逻辑和调用顺序，仅增加日志输出
- 确保不打印任何敏感信息（API key、headers 等）

### 变更文件

- `scripts/chat.py` - 增加两处调试输出：
  1. 在 `search()` 完成后，输出命中记忆的详细信息（序号、分数、内容截断）
  2. 在调用 `chat_completion()` 前，输出完整的 messages payload（JSON 格式）

### 使用示例

**终端输出示例：**

```
You> 我喜欢手冲咖啡

[debug][memory] hits=2
[debug][memory] #1 score=0.873 content=我家有一只11岁的柯基，名字叫丁丁，它非常喜欢在公园里奔跑...
[debug][memory] #2 score=0.420 content=为了记录：我住在26楼，每天早上都会喝一杯手冲咖啡，不加糖...

[debug][llm] messages payload:
[
  {
    "role": "system",
    "content": "你是一个智能助手 Jarvis。用户会向你提问或分享信息。\n请结合用户的本轮发言和相关记忆（如果有）给出简洁、有用的回复。\n如果相关记忆为空，仅基于用户本轮发言作答。"
  },
  {
    "role": "user",
    "content": "用户本轮发言：\n我喜欢手冲咖啡\n\n相关记忆（可能为0~K条）：\n1) [note] 我家有一只11岁的柯基，名字叫丁丁...（2025-11-10）\n2) [note] 为了记录：我住在26楼，每天早上都会喝一杯手冲咖啡，不加糖...（2025-11-10）\n\n请基于以上信息简洁作答。"

Jarvis> 根据您的记忆，您每天早上都会喝一杯手冲咖啡，不加糖。看来您对手冲咖啡有特别的喜好。

[store: ok] [search: hits=2] [llm: used]
```

```
[store: ok] [search: hits=0] [llm: used]
```

### 实现细节

**记忆命中调试输出：**
- 使用 `log.info()` 输出，级别为 INFO
- 格式：`[debug][memory] hits=N` 或 `[debug][memory] no hits`
- 每条记忆输出：`[debug][memory] #序号 score=分数 content=内容（截断至120字符）`
- score 保留 3 位小数，None 时显示为 "None"
- content 超过 120 字符时自动截断并添加 "..."

**LLM prompt 调试输出：**
- 在调用 `chat_completion()` 前构造 messages（与函数内部逻辑一致）
- 使用 `json.dumps(messages, ensure_ascii=False, indent=2)` 格式化输出
- 格式：`[debug][llm] messages payload:\n{JSON}`
- 仅打印 messages 列表（role + content），不包含任何敏感信息

### 验证要点

- REPL 能正常启动
- 输入一句话时，终端中能看到：
  - `[debug][memory] ...`（命中列表或 no hits）
  - `[debug][llm] messages payload: ...`（完整 prompt JSON）
- 原有的 `[store/search/llm]` 摘要输出保持不变
- 不打印任何敏感信息（API key、URL、headers 等）

## Update 2：完善记忆读写逻辑重构

### 目的与要点

- 完善「Jarvis ↔ OpenMemory」的记忆读写逻辑
- 重构大模型提示词构造部分，使其**非常清晰、集中、易于后期修改**
- 实现智能记忆策略：不是所有话都要记，由 LLM 判断是否值得长期记忆
- 打通时间戳字段，支持去重状态显示

### 变更文件

**新增模块：**
- `core/memory/classifier.py` - 消息分类模块（EXPLICIT_STORE / NEVER_STORE / CANDIDATE）
- `core/llm/prompt_builder.py` - LLM 提示词构造模块（集中、清晰、易改）
- `core/llm/response_parser.py` - LLM 返回解析模块（防御性解析）
- `core/memory/metadata_builder.py` - 记忆元数据构建模块（合并系统级和 LLM 提供的 tags/metadata）

**修改文件：**
- `scripts/chat.py` - 重构主流程（分类→搜索→LLM→落盘）
- `core/memory/client.py` - 修改 `search()` 解析时间戳，修改 `store()` 返回 `StoreResult`
- `core/memory/types.py` - 添加 `StoreResult`，扩展 `SearchResult` 和 `make_memory_item`
- `core/llm/deepseek.py` - 简化接口，接受已格式化的 `user_prompt`
- `scripts/memory_search.py` - 更新时间戳显示
- `utils/config.py` - 默认 `user_id` 设为 "lushan"

### 核心功能

**1. 消息分类（MemoryStoreDecision）**
- `EXPLICIT_STORE`: 显式要求记录（"为了记录：""帮我记一下"等）
- `NEVER_STORE`: 明显不需要记录（debug 命令、一次性问句）
- `CANDIDATE`: 不确定，由 LLM 判断

**2. LLM 提示词构造（集中、清晰、易改）**
- 系统 prompt 定义在 `BASE_SYSTEM_PROMPT` 常量中，包含完整的 JSON schema 说明
- 根据 policy 动态添加策略说明
- 用户 prompt 清晰组织：相关记忆（带时间）→ 用户发言 → 指令

**3. LLM 返回解析（防御性）**
- 支持从 markdown 代码块中提取 JSON
- JSON 解析失败时降级为原始文本
- 验证 content 长度、tags/metadata 格式

**4. 时间戳打通**
- `MemoryClient.search()` 解析 `last_seen_at` / `created_at`，格式化为 `YYYY-MM-DD HH:MM`
- `SearchResult.timestamp` 存储格式化时间字符串
- `scripts/memory_search.py` 和 REPL debug 输出显示时间

**5. 记忆写入（LLM 建议 + 去重状态）**
- 根据 LLM 返回的 `memories_to_store` 写入
- 合并系统级 tags（jarvis, user:xxx, channel:xxx）和 LLM 提供的 tags
- 合并系统级 metadata（source, autolog, recorded_at）和 LLM 提供的 metadata
- 解析 `deduplicated` 字段，显示 `[store: new]` 或 `[store: dedup]`
- EXPLICIT_STORE 的 fallback：JSON 解析失败时直接存储用户输入（去掉前缀）

### LLM 系统 Prompt 最终版本

```python
BASE_SYSTEM_PROMPT = """你是 Jarvis 的对话与记忆助手。每次对话你要做两件事：

1. 回复用户（reply 字段）
2. 如果本次对话中有值得长期记忆的事实或偏好，提炼为若干条记忆（memories_to_store 字段）

你必须返回一个严格的 JSON 对象，格式如下：

{
  "reply": "<你要回复给用户的话>",
  "memories_to_store": [
    {
      "content": "<适合写入记忆库的单句事实>",
      "tags": ["可选的标签数组"],
      "metadata": {
        "importance": 0.1~1.0,
        "event_date": "YYYY-MM-DD，可选",
        "valid_from": "ISO 时间字符串，可选",
        "valid_to": "ISO 时间字符串或 null，可选",
        "...": "你认为有用的其他字段"
      }
    }
  ]
}

注意：
- 一定要返回合法 JSON，不要包含多余的注释或解释文字。
- reply 应该是自然语言完整回复。
- memories_to_store 用于长期存储，只有在确实值得长期记忆时才添加。
- content 应该是简洁、独立的单句事实，避免冗长描述。
- tags 用于分类和检索，选择 1-5 个相关标签即可。
- metadata.importance 表示这条记忆的重要性（0.1=低，1.0=高）。
"""
```

### 时序说明

**完整流程：用户输入 → 分类 → 搜索 → 调 LLM → 落盘**

1. **用户输入**：读取用户输入文本
2. **消息分类**：调用 `classify_message_for_memory()` 判断策略（EXPLICIT_STORE / NEVER_STORE / CANDIDATE）
3. **检索记忆**：调用 `MemoryClient.search()` 检索相关记忆，解析时间戳
4. **构造提示词**：调用 `build_deepseek_prompt()` 构造系统 prompt 和用户 prompt
5. **调用 LLM**：调用 `chat_completion()` 获取原始 JSON 响应
6. **解析响应**：调用 `parse_deepseek_response()` 解析 JSON，提取 `reply` 和 `memories_to_store`
7. **写入记忆**：
   - 对每条 `memories_to_store`：
     - 调用 `build_final_tags_and_metadata()` 合并 tags/metadata
     - 调用 `make_memory_item()` 创建 `MemoryItem`
     - 调用 `MemoryClient.store()` 写入，获取 `StoreResult`
     - 根据 `status`（new/dedup）记录状态
   - EXPLICIT_STORE fallback：如果 JSON 解析失败或 `memories_to_store` 为空，直接存储用户输入（去掉前缀）
   - NEVER_STORE：忽略 LLM 返回的 `memories_to_store`
8. **输出摘要**：显示 `[store: new=2, dedup=1] [search: hits=3] [llm: used]`
